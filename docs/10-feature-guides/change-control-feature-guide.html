1. Module overview

ModuleChange Control
Pharma purposeControl + document any proposed change to an approved process, material, equipment, specification, or procedure. Drive impact assessment, multi-level approval, implementation, effectiveness verification.
Compliance basisICH Q10 §3.2.3 (change management) · 21 CFR 211.100 · 21 CFR 820.30 (medical device) · EU GMP Annex 15
Hawkeye routes/change-controls (frontend) · /api/universal/change-controls (backend)
Hawkeye model filebackend/src/models/ChangeControlModel.js
Tenant module flagmodules.CHANGE_CONTROL

Changes follow DRAFT → SUBMITTED → IMPACT_ASSESSMENT → UNDER_REVIEW (multi-step approval) → APPROVED → IMPLEMENTATION → VERIFICATION → CLOSED (or REJECTED / CANCELLED). The Wave 2 Regulatory Impact Classifier suggests US (CBE-30 / PAS-NDA) + EU (Type IA/IB/II Variation) routing from the change description.

4.2 + New Change Request dialog

What it doesAuthor a new change request.
Menu / route/change-controls · top-right '+ New Change Request' button
Available to rolescreateRoles
Backend APIPOST /api/universal/change-controls

Click-by-click

  1. CLICK Click '+ New Change Request'
    → Dialog opens
  2. TYPE Fill title (1 sentence)
    → required
  3. TYPE Fill description (full narrative)
    → required
  4. CLICK Pick changeType (PROCESS / MATERIAL / EQUIPMENT / DOCUMENTATION / FACILITY / SUPPLIER / SPECIFICATION / OTHER)
    → required
  5. CLICK Pick riskLevel (LOW / MEDIUM / HIGH / CRITICAL)
    → required
  6. TYPE (Optional) Fill affectedProducts[] + affectedMarkets[]

  7. CLICK (Optional) Add approvalSteps[] now or after submit

  8. CLICK Click 'Save'
    → Row appears with status=DRAFT; changeNumber=CCR-YYYY-NNNN auto-generated

4.5 Approve / reject step

What it doesApprover acts on a pending step. All APPROVED → status=APPROVED; any REJECTED → status=REJECTED.
Menu / routeChange detail · 'Approve' button (visible to assigned approver)

Backend APIPOST /api/universal/change-controls/:id/approval

  3. TYPE Fill comments (recommended)
    → stored on the step
  4. CLICK Submit

4.6 Verify effectiveness (IMPLEMENTATION/VERIFICATION → CLOSED)

What it doesAfter the change is implemented, verify it achieved its intent.
Menu / routeChange detail · 'Verify Effectiveness' button
Available to rolescreateRoles
Backend APIPOST /api/universal/change-controls/:id/verify-effectiveness

Click-by-click

  1. CLICK Click 'Verify Effectiveness'
    → Drawer with verificationNotes + effectivenessCheck + effective boolean
  2. TYPE Fill verificationNotes
    → required
  3. TYPE Fill effectivenessCheck (the criterion)
    → required
  4. CLICK Tick effective=true (or false)
    → true → status=CLOSED; false → status=IMPLEMENTATION (re-work)
  5. CLICK Submit
    → Status updates

9. Known gaps + roadmap

  • Mandatory e-sig per approval step — Wire /api/electronic-signatures/sign for each approve action. HIGH
  • Auto-link to document revision when changeType=DOCUMENTATION — Today linkedDocumentIds is manual. Auto-create + link a new SOP rev. MEDIUM
  • Auto-trigger training when implementation completes — If linkedDocumentIds includes a SOP that publishes, training auto-assign already fires. Confirm cross-link. LOW
  • AI suggested approver list — Based on changeType + affectedProducts, suggest approvalSteps[] from a tenant policy table. LOW
