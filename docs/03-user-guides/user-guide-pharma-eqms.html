4. Module Guide: Document Control

Manages the lifecycle of all controlled documents — SOPs, policies, work instructions, specifications, protocols.

Figure 2: Document Control page — filter by status and document type

Document Lifecycle

DRAFT→ UNDER_REVIEW→ APPROVED→ EFFECTIVE→ SUPERSEDED or WITHDRAWN

How to Use

1
Create a document
Click "+ New Document". Enter title, type (SOP/Policy/Work Instruction/etc.), scope. Auto-assigns DOC-YYYY-NNNN number.
2
Route for approval
Add approval steps (e.g. Department Head → QA Manager). Each approver reviews and signs electronically.
3
Publish
Once approved, click "Publish" to make the document EFFECTIVE. Set the effective date and review due date.
4
Supersede (new version)
When a revision is needed, click "Supersede". A new version is created as DRAFT; the current version moves to SUPERSEDED.
Training trigger: If requiresTrainingOnUpdate is enabled, publishing a new version automatically triggers a training assignment for all users associated with that document's department.
Document TypesAuto-numbering
SOP, Policy, Work Instruction, Form, Specification, Protocol, Report Template, Guideline, Regulatory Submission, CustomDOC-2026-0001, DOC-2026-0002, ...

Change Control Lifecycle

DRAFT→ SUBMITTED→ IMPACT_ASSESSMENT→ UNDER_REVIEW→ APPROVED→ IMPLEMENTATION→ VERIFICATION→ CLOSED

How to Use

1
Initiate change request
Select type (Process/Equipment/Material/Document/System), risk level (Low/Medium/High/Critical), describe the change and justification.
2
Impact assessment

3
Approve & implement

4
Verify & close
Verify the change achieved its intended outcome. Conduct effectiveness check. Close the CCR.

6. Module Guide: Deviation / Non-Conformance

Figure 4: Deviation / NC Manager — investigate, assess impact, dispose, link to CAPA

Deviation Lifecycle

REPORTED→ UNDER_ASSESSMENT→ UNDER_INVESTIGATION→ PENDING_DISPOSITION→ PENDING_CAPA_DECISION→ CAPA_REQUIRED→ PENDING_CLOSURE→ CLOSED

How to Use

1
Report deviation
Anyone can report. Classify as PLANNED or UNPLANNED. Set category (Process/Equipment/Material/Environmental/Laboratory/etc.) and classification (Critical/Major/Minor). Auto-numbers: DEV-2026-XXXX.
2
Assess impact
Evaluate: product quality impact, patient safety impact, batch disposition (Release/Reject/Rework/Quarantine), regulatory impact.
3
Investigate root cause
Select RCA method (5-Why, Fishbone, Fault Tree, Pareto). Document root cause and root cause category.
4
Dispose & decide CAPA
Set batch disposition. Decide if CAPA is required. Link to existing CAPA records if applicable.

11. Module Guide: External Supplier Audit

As a supplier, you receive audits from your customers. The 8-phase workflow runs the same as described in the Pharma Audit guide. Your role is to:

  1. Receive and accept intimation letters
  2. Complete pre-audit and execution questionnaires
  3. Provide evidence documents
  4. Submit CAPA plans for findings
  5. Track CAPA implementation to closure

12. Module Guide: Internal Audit

Use the same 8-phase audit workflow for self-inspections. Create an audit where your organization is both the buyer (initiator) and the supplier (site being audited). Assign an internal auditor.

Assessment Type: Select "Internal" or "Self" when creating the audit. This uses the same templates and workflow but flags the audit as an internal self-inspection for regulatory purposes (EU GMP Chapter 9, ISO 9001 clause 9.2).

13. Cross-Module Integration: The Quality Event Cascade

The power of a full EQMS is that modules are interconnected. A single quality event can cascade across multiple modules. Here's a real-world example:

Scenario: Temperature excursion in cold storage

1
Deviation Reported
Warehouse operator reports temperature excursion → DEV-2026-0001 created in /nonconformance
↓
2
Impact Assessed
QA assesses: batch quarantined, root cause = HVAC relay failure (Equipment)
↓
3
Equipment Record Updated
Go to /asset-management. HVAC unit flagged as OUT_OF_SERVICE. Calibration overdue. Relay replaced → new calibration recorded.
↓
4
Customer Complaint Logged
Customer reports quality concern → CMP-2026-0001 linked to deviation DEV-2026-0001 in /complaint-manager
↓
5
Change Control Raised
Engineering raises CCR-2026-0001: "Add dual-relay failover to cold storage HVAC" in /change-controls
↓
6
SOP Updated
Preventive Maintenance SOP revised → new version DOC-2026-0005 published in /document-control
↓
7
Training Assigned
Warehouse team assigned retraining on updated PM SOP in /training
↓
8
Management Review
All data flows into quarterly review. QMS adequacy assessed. Action items tracked in /management-review
Single source of truth: Every record links back — deviation → complaint → equipment → change → document → training → management review. No spreadsheets, no duplicate data, complete traceability for regulatory inspections.

14. AI Co-Pilot — Inline Assists, Agent Flows, Predictive Insights

HawkEye ships an AI layer built on the principle of "AI proposes, humans decide". Every AI-drafted output is reviewable, editable, and must be e-signed before it becomes record. Every AI decision is captured in the main AuditTrail with prompt-hash, retrieval set IDs, model version, and confidence — FDA inspectors can reconstruct any recommendation.

Smoke-tested end-to-end: 11/12 AI endpoints pass against a local backend with the Novex Pharma tenant logged in as qa.head@novex-pharma.demo. The single skip is the platform correctly refusing low-confidence ungrounded output for an under-specified deviation scaffold — the grounding gate doing its job.

14.3 Persona walkthrough — who uses what

QA Specialist (Kenji Tanaka) — owns deviation investigation + CAPA

  1. Opens a deviation in /nonconformance; clicks "Scaffold 5-why with AI". AI proposes 5 probing questions + 3 follow-up questions to raise on-site + 6M categorisation. Investigator edits, fills probable answers, e-signs.
  2. Once root cause confirmed, opens a CAPA at /buyer/capas and clicks "Draft RCA with AI". AI returns full structured RCA: 5-why, fishbone, corrective + preventive actions with owners + due dates, effectiveness check, severity, regulatory clauses.
  3. Reviews each section, edits as needed, e-signs the CAPA. Outcome (accepted / edited / rejected) is logged via /api/ai/decisions/outcome; this feeds the active-learning loop.
  4. Inline predictive badge shows P(on-time) + P(effective) for the CAPA. If P(on-time) < 60%, considers shortening due-date or escalating ownership.
Time saving: CAPA RCA cycle in our smoke test took ~14 seconds end-to-end (Wave 1 endpoint round-trip). Manual drafting typically takes 30-90 minutes. Veeva pilots cite 75% cycle reduction; Hawkeye targets 40%+.

Lead Internal Auditor (Maria Santos) — runs internal + supplier audits

  1. Before an audit, opens Supplier Risk Dossier for the auditee. AI compiles FDA 483s, warning-letter citations, prior audit history, open CAPAs into a one-page risk score (LOW/MEDIUM/HIGH/CRITICAL) with section-by-section narrative + citations.
  2. During the live audit (/audits/[id]/questionnaire), enters supplier responses. After each response, the inline Follow-up Suggester proposes 2-3 probing questions. Maria clicks "Add" to insert any she likes.
  3. When she has enough evidence for a finding, clicks "Draft observation". AI returns a structured draft: title, factual narrative in regulatory voice, severity (minor/major/critical), CAPA-worthy flag, mapped 21 CFR / ICH clauses, evidence citations.
  4. Maria edits, sets her own severity if she disagrees with AI, e-signs the observation. AI never publishes — only she does.
Note: Every AI step is logged. If Maria is challenged on a finding during the closing meeting, she can show the auditor exactly which evidence + which regulatory clause AI cited.

Audit Program Manager (Priya Nair) — strategic + agentic

  1. Opens the Agent Console. Types a goal in plain English: "Summarise the current status of open CAPAs for this tenant and flag any > 30 days late."
  2. AI drafts a multi-step plan: capas.open_list → filter by age → group by owner → return summary. Priya reviews each step, can edit inputs, removes any she doesn't want.
  3. Approves with the e-sig checkbox. Runtime executes step-by-step within budget (max 8 steps, 20K tokens, 180s). Each step writes to AuditTrail.
  4. Result is shown inline. Steps with write side-effects require an additional e-sig ticket — read-only steps execute immediately.

QC Lab Lead (Aisha Patel) — auditee + deviation reporter

  1. Files an OOS deviation. Uses the same 5-why scaffolder as the QA Specialist — but with QC-specific context (test method, lot, retest results).
  2. When responding to an audit questionnaire section assigned by Production Head, sees AI-suggested follow-ups appearing inline. (These appear to the auditor's UI, not Aisha's.)

Maintenance Engineer (Lars Nilsson) — IoT equipment

  1. An external MQTT bridge POSTs telemetry to /api/ai/iot/telemetry for every connected piece of equipment.
  2. If a measurement falls outside the equipment's calibration spec, the platform auto-opens a deviation (Lars or QC reviews).
  3. Lars opens an equipment page and sees MTBF prediction (mean-time-between-failures heuristic from variance + excursion rate). If < 45 days, schedules preventive maintenance proactively.

Head of QA (James Thompson) — operations + drift

  1. Opens the Drift Monitor Dashboard daily. Sees grounded-rate, user-acceptance, p95 latency, tool-failure rate per AI feature — current vs 7-day baseline.
  2. If any metric drifts beyond threshold (default: grounded-rate down 5pp, acceptance down 10pp, latency up 25%), a DriftAlert is raised. James reviews and either acknowledges or pauses the feature behind a flag while the AI team investigates.
  3. Reviews open Signal Alerts from the deviation signal detector. Each alert shows a cluster of similar deviations (shared equipment / lot / SOP) with z-score vs 6-month baseline. Closes as True positive (opens systemic CAPA) or False positive (feeds the model with feedback).

VP Quality (Dr Elena Vasquez) — review + governance

  1. For any AI-drafted record (CAPA, observation, MRM input), can drill into the AI decision log to see exactly: who triggered, prompt hash, model + version, retrieval set, confidence, and human disposition.
  2. For GxP-paranoid context (e.g. patient-safety CAPAs), can switch the tenant to on-prem LLM mode via POST /api/ai/onprem/endpoint — same gateway API, same audit trail, but data never leaves the tenant VPC.

15. Audit AI Agents — Deep-Dive

Dedicated agents that drive the audit lifecycle. Every agent is a thin composition over the Wave 1-3 primitives (LLM gateway, grounded-generation runtime, tool-calling runtime, AI audit trail) plus one new shared service: PublicDataFusion + EntityResolution. Routes mounted under /api/ai/audit-agents.

Provenance tagging everywhere: every data point the UI renders carries a source tag — tenant (your registered data), openFDA / fdaWarningLetter (public regulatory), pharmaCompass (commercial, opt-in), or inferred (AI-derived from the above). Users never confuse public signals with tenant-registered records.

15.1 The four agents

AuditPrepAgent — risk-weighted questionnaire

Given supplier + product + scope, the agent:

  1. Resolves the supplier entity (known_tenant / public_only / ambiguous / unknown).
  2. Fetches past audit findings for this supplier from tenant DB.
  3. Fetches openFDA drug registrations, recalls, warning letters for the supplier name.
  4. Loads a baseline questionnaire template (default PSCI SAQ).
  5. LLM risk-weights each section — adds extra questions tied to specific FDA signals, drops low-value sections, cites every new question to its source (finding:ID, recall:ID, wl:URL).

Endpoint: POST /api/ai/audit-agents/prepare-questionnaire

Frontend: components/ai/audit-agents/AuditPrepAgent.tsx

Smoke-tested: 6 sections, 4 high-risk signals flagged, confidence 0.90, via free Gemini 2.5 Flash-Lite.

AuditAutofillAgent — form filler from library docs

For any form schema (list of {name, label, type, hint} field specs):

  1. Uses library chunks (caller pre-OCRs PDFs; Tesseract path exists) or falls back to tenant profile (SupplierProfile + Site + Product).
  2. For each field: grounded-gen call → extracts best-fit value + confidence + citation.
  3. UI shows per-field confidence chip; "Apply all confident" only fills fields above threshold.

Endpoint: POST /api/ai/audit-agents/autofill-form

Frontend: components/ai/audit-agents/AutofillButton.tsx — drop-in for any MUI form.

SupplierIntelAgent — tenant + public dossier

Combines EntityResolutionService (fuzzy match to registered suppliers) with PublicDataFusion (openFDA + FDA warning letters) in one unified view. Explicitly labels what's your data vs what's public.

Endpoint: POST /api/ai/audit-agents/supplier-intel

Frontend: components/ai/audit-agents/SupplierIntelCard.tsx — shows verdict chip, tenant card (blue), public card (orange).

Pharma Compass adapter is present but disabled by default — enable by setting PHARMA_COMPASS_API_KEY. Their ToS does not permit open scraping.

AuditReportAgent — assemble + hash

For an audit ID, assembles the full report (scope + findings + CAPAs + evidence), writes a grounded exec summary + recommendations with per-claim citations, and emits a standalone HTML with a SHA-256 integrity hash. The hash makes any future ledger-anchor (Sigstore Rekor, Polygon, etc.) a 10-line add-on.

Endpoint: POST /api/ai/audit-agents/assemble-report

Frontend: components/ai/audit-agents/AuditReportGenerator.tsx — one-click assemble, preview in new tab, download HTML.
